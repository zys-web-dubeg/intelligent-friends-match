## 1. 项目概述

### 1.1 业务定位

本项目是一个**基于大语言模型（LLM）的智能社交匹配平台后端服务**，提供：

- **AI 角色对话**：多个领域专属 AI 助手（就医、购物、恋爱、学习、伙伴匹配）
- **智能匹配引擎**：基于 MBTI + Embedding 语义特征的多维匹配系统
- **团队协作**：AI 辅助的团队聊天与组队匹配
- **业务子系统**：预约挂号、购物订单、用户认证、统计分析

### 1.2 核心架构风格

| 维度 | 说明 |
|------|------|
| 架构模式 | 分层架构（Controller → Service → Mapper → MySQL）+ AI Service 层 |
| AI 集成 | LangChain4j `@AiService` 注解驱动的声明式 AI 服务 |
| 流式响应 | Project Reactor `Flux<String>` 流式 SSE 输出 |
| 通信方式 | REST API + WebSocket 实时通信 |
| 认证方式 | JWT（HS256, 24h 过期）+ 登录验证码 |

## 4. AI 服务层规格

### 4.1 AI 服务架构

采用 **LangChain4j `@AiService` 声明式编程模型**，核心组件：

```
┌─────────────────────────────────────────────────────────┐
│                    @AiService 接口                         │
│  (wiringMode = EXPLICIT, 显式注入所有依赖)                │
├─────────────────┬─────────────────┬─────────────────────┤
│  ChatLanguageModel│ ChatMemory(Provider)│  ContentRetriever│
│  / Streaming     │                  │  (RAG EmbeddingSearch)│
├────────┬────────┴────────┬────────┴─────────────────────┤
│ Tools  │  SystemMessage   │  @UserMessage / @MemoryId   │
└────────┴─────────────────┴─────────────────────────────┘
```

**说明：**
- `@AiService` 负责将 LLM 调用、记忆管理、RAG、工具调用绑定为一个完整的 AI 服务
- `wiringMode = EXPLICIT` 模式要求开发者显式配置聊天模型、记忆和工具
- `@SystemMessage(fromResource = "...")` 从外部 `.txt` 文件加载系统提示词
- `@MemoryId` 标记对话 ID 参数，用于多会话隔离
- `@Tool` 注解方法自动注册为 AI 可调用的函数

## 5. 多维匹配引擎规格

### 5.1 评分模型架构

匹配引擎位于 `com.ithuangma.java.ai.langchain4j.matching` 包，由 `MultiDimensionMatcher` 统一调度，`MbtiCompatibility` 提供 MBTI 维度评分。

### 5.2 团队匹配维度与权重

| 维度 | 权重 | 计算方法 |
|------|------|----------|
| MBTI 兼容性 | 25% | E/I 互补（0.85 vs 0.50）、S/N 相同（0.80 vs 0.45）、T/F 互补（0.75 vs 0.50）、J/P 互补（0.75 vs 0.50）→ 四维平均 |
| 兴趣标签语义 | 40% | 用户画像文本 vs 队伍约束文本的 Embedding 余弦相似度 |
| 性格特征 | 20% | 用户性格特征 vs 队伍约束的 Embedding 余弦相似度 |
| 沟通风格 | 15% | 用户沟通风格 vs 队伍约束的 Embedding 余弦相似度 |

### 5.3 相似用户匹配维度与权重

| 维度 | 权重 | 计算方法 |
|------|------|----------|
| MBTI 兼容性 | 20% | 四维相同比例（相同维度数/4） |
| 兴趣标签语义 | 40% | 两人画像 Embedding 余弦相似度 |
| 兴趣爱好关键词 | 20% | 两人兴趣描述的 Embedding 余弦相似度 |
| 沟通风格 | 20% | 两人沟通风格的 Embedding 余弦相似度 |

### 5.4 伙伴匹配（按类型差异化）

| 匹配类型 | MBTI 策略 | MBTI权重 | 嵌入权重 | 兴趣权重 | 沟通权重 |
|----------|-----------|----------|----------|----------|----------|
| date（恋爱） | 互补评分 | 25% | 25% | 25% | 25% |
| study/hobby/sports/game/friend | 相似评分 | 15% | 35% | 30% | 20% |

### 5.5 余弦相似度计算

```java
score = sum(v1[i] * v2[i]) / (sqrt(sum(v1[i]^2)) * sqrt(sum(v2[i]^2)))
```

## 6. 向量数据库与 RAG

### 6.1 向量数据库架构

```
┌─────────────────────────────────────────────┐
│               EmbeddingStore                   │
├───────────────────┬─────────────────────────┤
│   Pinecone        │  InMemoryEmbeddingStore  │
│   (生产环境)       │  (降级备用)              │
├───────────────────┴─────────────────────────┤
│   索引: friendmatch-index                     │
│   命名空间: friendmatch-namespace             │
├─────────────────────────────────────────────┤
│   连接策略: 最多重试3次, 指数退避(2s/4s/6s)    │
│   全部失败后自动降级为内存向量库               │
└─────────────────────────────────────────────┘
```

### 6.2 Embedding 模型

| 属性 | 值 |
|------|-----|
| 模型 | text-embedding-v3（DashScope 通义千问） |
| API Key | 环境变量 `DASH_SCOPE_API_KEY` |
| 维度 | 1024（text-embedding-v3 默认） |

### 6.3 向量库元数据

| 字段 | 说明 |
|------|------|
| `type` | 实体类型：`team` / `user` |
| `teamId` | 队伍ID（type=team 时） |
| `user_id` | 用户ID（type=user 时） |

### 6.4 向量索引初始化

应用启动时通过 `CommandLineRunner` 执行：

| 初始化器 | 索引对象 | 元数据 |
|----------|----------|--------|
| TeamVectorInitializer | 所有 TeamProfile 的 personalityConstraint | {teamId, type:"team"} |
| UserProfileVectorInitializer | 所有 UserProfile 的完整画像文本 | {user_id, type:"user"} |

**幂等策略**：已索引的记录跳过（通过数据库记录标记判断）。

## 7. 聊天记忆系统

### 7.1 架构

```
┌─────────────────────────────────────────────────────┐
│                  用户请求 (@MemoryId)                 │
│                        │                             │
│              ┌─────────┴─────────┐                   │
│              │  ChatMemoryStore  │                   │
│              │ (MongoDB实现)      │                   │
│              ├───────────────────┤                   │
│              │ chat_sessions 集合 │                   │
│              │ - sessionId (PK)  │                   │
│              │ - messages[]      │                   │
│              │ - participants[]  │                   │
│              │ - userId / aiId   │                   │
│              └───────────────────┘                   │
└─────────────────────────────────────────────────────┘
```

### 7.2 MongoDB 集合

| 集合名 | 用途 | 主要字段 |
|--------|------|----------|
| `chat_sessions` | 聊天会话 | sessionId, messages[], participants[], teamId, userId, aiId, lastActiveTime |
| `chat_messages` | 聊天消息（辅助） | messageId, content, memoryId |

### 7.3 记忆类型

| 类型 | 组件 | 存储 | 容量 | 使用场景 |
|------|------|------|------|----------|
| 滑动窗口 | MessageWindowChatMemory | 内存 | 10条 | MemoryChatAssistant |
| MongoDB 持久化 | ChatMemoryProvider + MongoChatMemoryStory | MongoDB | 10-50条 | 各 AI Agent |
| 手动管理 | MongoTemplate 直接操作 | MongoDB | - | 团队聊天 |

### 7.4 消息类型转换

MongoChatMemoryStory 在 LangChain4j `ChatMessage` 和 MongoDB 文档之间做双向转换：

| LangChain4j 类型 | SenderType | 存储方式 |
|------------------|-----------|----------|
| UserMessage | HUMAN | content + senderId/senderType |
| AiMessage | AI | content + 含 toolExecutionRequests 的 meta |
| SystemMessage | SYSTEM | content |
| ToolExecutionResultMessage | TOOL | content + 含 toolName/toolExecutionId 的 meta |

## 8. WebSocket 实时通信

### 8.1 端点

```
ws://host:port/websocket/team/{teamId}/{userId}?token={jwt}
```

### 8.2 消息协议

**客户端 → 服务器：**

```json
{
  "type": "MESSAGE" | "AI_TRIGGER",
  "content": "消息内容",
  "teamId": "队伍ID"
}
```

**服务器 → 客户端：**

```json
// 人类消息广播
{
  "type": "MESSAGE",
  "senderType": "HUMAN",
  "senderId": "用户ID",
  "senderUserId": "用户ID",
  "content": "消息内容",
  "timestamp": 1718524800000
}

// AI消息广播
{
  "type": "AI_MESSAGE",
  "senderType": "AI",
  "senderId": "ai_assistant",
  "content": "AI回复内容",
  "timestamp": 1718524800000
}
```

### 8.3 AI 参与判定

`shouldAIParticipate()` 方法基于关键词匹配判断是否触发 AI 回复。匹配以下条件之一即触发：

- 英文关键词（小写）：`ai`, `help`, `suggest`, `recommend` 等
- 中文关键词：`帮忙`, `帮助`, `建议`, `推荐`, `应该`, `多少`, `什么`, `怎么`, `为什么`, `哪个`, `请教`, `告诉`, `知道`, `好不好`, `对吗`, `正确`, `问题`, `方法`, `学习` 等
- 问号：中文问号 `？` 或英文问号 `?`

### 8.4 连接管理

- 连接存储：`ConcurrentHashMap<"teamId_userId", WebSocket>` + `CopyOnWriteArraySet` 全量集合
- 认证：从 query string 提取 JWT token，验证通过后建立连接
- 广播：按 teamId 过滤，向同一队伍的所有在线成员发送消息

## 9. 认证与安全

### 9.1 JWT 认证

| 参数 | 值 |
|------|-----|
| 签名算法 | HS256 |
| 密钥 | 12位字符串（硬编码于 JwtUtils） |
| 过期时间 | 24小时 |
| Token 位置 | HTTP Header `Authorization: Bearer <token>` |
| 拦截范围 | `/api/**`（排除登录/注册/验证码/Swagger） |

### 9.2 验证码

- 端点：`GET /captcha`
- 类型：图片验证码
- 存储：Redis，关联到验证会话

### 9.3 密码安全

- 算法：MD5 + 固定盐值 `"songsong"`
- 注册校验：用户名特殊字符校验、手机号/邮箱格式正则

## 13. Redis 缓存策略

### 13.1 缓存键设计

| 缓存键模式 | 用途 | TTL |
|-----------|------|-----|
| `team:recommendation:user:{userId}` | 队伍推荐缓存 | 30分钟，异步刷新 |
| `user:similar:{userId}` | 相似用户缓存 | 30分钟，异步刷新 |

### 13.2 缓存更新策略

- **Cache-Aside + 异步刷新**：缓存命中时立即返回，同时异步重新计算并更新缓存
- **缓存失效**：反序列化失败时主动删除损坏缓存
- **序列化**：`GenericJackson2JsonRedisSerializer`

## 14. 分布式锁

使用 Redisson 实现分布式锁，防止并发加入队伍：

```java
RLock lock = redissonClient.getLock("team_join_lock:" + teamId + ":" + userId);
// 等待5秒，持有10秒后自动释放
boolean acquired = lock.tryLock(5, 10, TimeUnit.SECONDS);
```

**双重检测模式（Double-Check Locking）：**
1. 第1检（无锁）：快速失败，避免不必要的锁竞争
2. 第2检（有锁）：防止第1检与加锁之间被其他线程插入

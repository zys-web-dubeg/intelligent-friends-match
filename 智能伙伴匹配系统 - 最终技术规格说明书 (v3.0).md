### 智能伙伴匹配系统 - 最终技术规格说明书 (v3.0)

#### 一、核心架构与存储选型

本系统采用混合存储架构，以平衡关系数据的严谨性与非结构化数据的灵活性。

- **关系型数据库 (MySQL 8.0+)**
    - **职责**：存储强一致性的业务数据。包括用户账号、队伍信息、成员关系、权限配置等。
    - **定位**：系统的“骨架”。
- **向量数据库 (Pinecone)**
    - **职责**：存储高维 Embedding 向量。包括用户画像向量、AI人设向量、队伍主题向量。
    - **定位**：系统的“直觉”，负责软匹配和推荐。
- **文档数据库 (MongoDB 6.0+)**
    - **职责**：存储海量的、非结构化的对话记录。包括私聊历史、群聊记录、AI 长期记忆上下文。
    - **定位**：系统的“海马体”（长期记忆），负责为 LangChain4j 提供 `ChatMemoryStore` 实现。

#### 二、详细数据库设计

**1. MySQL：业务骨架**

| 表名                   | 关键字段                                                     | 说明                                      |
| ---------------------- | ------------------------------------------------------------ | ----------------------------------------- |

**2. MongoDB：对话记忆 (核心新增)**

我们需要两个核心集合来分别处理“即时聊天流”和“长期记忆”。

**A. 对话会话集合 (**`**chat_sessions**`**)**
用于存储具体的聊天记录，类似微信的聊天记录。

```json
{
  "_id": "ObjectId",
  "session_id": "sess_1001",           // 会话唯一标识
  "type": "GROUP",                     // 类型：PRIVATE(私聊) 或 GROUP(群聊)
  "participants": ["user_A", "ai_B"],  // 参与者列表
  "team_id": "team_555",               // 如果是群聊，关联队伍ID
  "messages": [                        // 消息数组（ capped collection 或 分页存储）
    {
      "sender_id": "user_A",
      "sender_type": "HUMAN",
      "content": "今天心情不好",
      "timestamp": "ISODate('2026-05-03T10:00:00Z')",
      "meta": {"emotion": "sad"}
    },
    {
      "sender_id": "ai_B",
      "sender_type": "AI",
      "content": "发生什么事了？愿意跟我说说吗？",
      "timestamp": "ISODate('2026-05-03T10:00:02Z')"
    }
  ],
  "last_active_time": "ISODate('2026-05-03T10:00:02Z')"
}
```

**B. 长期记忆摘要集合 (**`**conversation_memories**`**)**
用于 LangChain4j 进行 RAG 检索。因为聊天记录会无限增长，不能每次都把全量记录发给 LLM，所以需要定期将 MongoDB 中的对话压缩成摘要存入此处，并向量化后存入 Pinecone。

```json
{
  "_id": "ObjectId",
  "memory_id": "mem_user_A_ai_B",      // 关联用户和AI
  "summary": "用户A最近因工作压力大感到焦虑，喜欢被倾听。", 
  "vector_id": "pinecone_vec_999",     // 对应 Pinecone 中的向量ID
  "created_at": "ISODate(...)"
}
```

#### 三、LangChain4j + MongoDB 集成方案

我们将实现一个自定义的 `MongoChatMemoryStore`，让 LangChain4j 自动通过 MongoDB 读写历史。

**1. 依赖引入**

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-mongodb</artifactId>
</dependency>
<dependency>
    <groupId>dev.langchain4j</groupId>
    <artifactId>langchain4j-spring-boot-starter</artifactId>
</dependency>
```

**2. 核心代码实现**
这是一个实现了 LangChain4j 接口的 MongoDB 存储类，它负责将内存中的对话持久化到 Mongo。

```java
@Component
public class MongoChatMemoryStore implements ChatMemoryStore {

    @Autowired
    private MongoTemplate mongoTemplate;

    private static final String COLLECTION_NAME = "chat_sessions";

    @Override
    public List<ChatMessage> getMessages(Object memoryId) {
        // 1. 从 MongoDB 查询该 memoryId (即 sessionId) 的最新文档
        Query query = new Query(Criteria.where("session_id").is(memoryId))
                .with(Sort.by(Sort.Direction.DESC, "last_active_time"))
                .limit(1);
        
        ChatSession session = mongoTemplate.findOne(query, ChatSession.class, COLLECTION_NAME);
        
        if (session == null || session.getMessages() == null) {
            return Collections.emptyList();
        }

        // 2. 将 JSON 格式的消息转换为 LangChain4j 的 ChatMessage 对象
        return session.getMessages().stream()
                .map(msg -> {
                    if ("HUMAN".equals(msg.getSenderType())) {
                        return UserMessage.from(msg.getContent());
                    } else {
                        return AiMessage.from(msg.getContent());
                    }
                })
                .collect(Collectors.toList());
    }

    @Override
    public void updateMessages(Object memoryId, List<ChatMessage> messages) {
        // 3. 将新的对话消息追加写入 MongoDB
        // 注意：生产环境建议只追加增量，这里为了演示简化逻辑
        Query query = new Query(Criteria.where("session_id").is(memoryId));
        Update update = new Update()
                .set("messages", convertToDbFormat(messages)) // 转换格式
                .set("last_active_time", new Date());
        
        mongoTemplate.upsert(query, update, ChatSession.class, COLLECTION_NAME);
    }
    
    // ... 辅助转换方法省略
}
```

**3. AI 服务调用**
配置好 Store 后，你的 AI 服务就可以自动拥有“记忆”了：

```java
@Service
public class AssistantService {

    // 注入上面定义的 Store
    @Autowired
    private MongoChatMemoryStore memoryStore;

    @Autowired
    private ChatLanguageModel chatModel; // LangChain4j 模型

    public String chat(String sessionId, String userMessage) {
        // LangChain4j 会自动调用 memoryStore.getMessages(sessionId) 获取历史
        // 并在回复后自动调用 memoryStore.updateMessages(...) 保存新记录
        
        return AiServices.builder(Assistant.class)
                .chatLanguageModel(chatModel)
                .chatMemoryProvider(memoryId -> MessageWindowChatMemory.withMaxMessages(10)) // 限制窗口大小
                .build()
                .chat(userMessage);
    }
}
```

#### 四、核心业务流程：加入队伍的完整闭环

结合 MongoDB，现在的流程更加完整：不仅判断能不能进，还能在进之后立刻“读懂”这个队伍的历史氛围。

**接口**: `POST /api/v1/teams/{teamId}/join`

**逻辑步骤：**

1. **准入判定 (MySQL + Pinecone)**
    - 检查 `team_profile` 表的 `access_level`。
    - 若是审核制，计算用户向量与队伍向量的相似度（Pinecone）。
    - 若分数达标，执行步骤 2。
1. **建立连接 (MySQL + MongoDB)**
    - 在 `team_member_relation` 插入记录。
    - **初始化会话**：在 MongoDB 的 `chat_sessions` 集合中，为该用户在队伍中创建一个初始会话文档（或者激活旧的会话文档）。
        - *目的*：确保用户一进入就能看到之前的聊天记录（如果队伍设定是公开历史记录的话）。
1. **AI 欢迎仪式 (LangChain4j + MongoDB)**
    - 触发后台任务。
    - **读取上下文**：AI 读取 MongoDB 中该队伍最近的 10 条消息（了解当前大家在聊什么）。
    - **生成欢迎语**：
    - **推送**：将生成的欢迎语写入 MongoDB，并通过 WebSocket 推送到前端。

#### 六、运维与成本优化建议

1. **MongoDB TTL 索引**
    - 对于临时会话或测试数据，可以设置 TTL 索引自动过期，节省存储空间。
    - *但在生产环境的主聊天记录上不要开启，这是用户的资产。*
1. **Pinecone 成本控制**
    - 不要把所有聊天记录都向量化存入 Pinecone。
    - **策略**：只对“重要事件”（如：用户表达了强烈情绪、达成了某个成就、发生了争吵）进行摘要并向量化。普通的“早安晚安”只存 MongoDB，不进 Pinecone。
1. **冷热分离**
    - **热数据**（最近 7 天的对话）：存在 MongoDB 主库，保证读写速度。
    - **冷数据**（7 天前的对话）：可以归档到 MongoDB 的冷集合或廉价的对象存储（S3/OSS）中，减少主库压力。
